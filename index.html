vcLog

Ah, the world of changes. Always changing things are. And keeping track of changes can be quite the challenge. Thankfully we have version control systems to make the chore a bit easier. But review these changes and we find our output options limited. That's were vcLog comes in.

vcLog can provide a changelogs in a variety of uniform formats for different SCMs. If we ask vcLog for help will see it's commandline interface is fairly self-explanitory.

  Usage: vclog [TYPE] [FORMAT] [OPTIONS] [DIR]
   
  OUTPUT TYPE (choose one):
      -l, --log, --changelog           changelog (default)
      -r, --rel, --history             release history
      -b, --bump                       display a bumped version number
      -c, --current                    display current version number
   
  FORMAT (choose one):
          --gnu                        GNU standard format (default)
          --xml                        XML format
          --yaml                       YAML format
          --json                       JSON format
          --html                       HTML micro-like format
          --rdoc                       RDoc format
      -m, --markdown                   Markdown format
   
  OTHER OPTIONS:
          --title               document title, used by some formats
      -e, --extra                      provide extra output, used by some formats
      -v, --version <NUM>              current version to use for release history
          --style [FILE]               provide a stylesheet name (css or xsl) for xml and html formats
          --id                         include revision ids (in formats that normally do not)
      -o, --output [FILE]              send output to a file instead of stdout
   
  STANDARD OPTIONS:
          --debug                      show debugging infromation
      -h, --help                       display this help information
  </pre>

  <p>Without any options, vcLog will output a standard GNU ChangeLog format.</p>

  <pre>
  $ vclog

  2009-10-23   7rans <transfire@gmail.com>

          * update pom metadata


  2009-10-12   7rans <transfire@gmail.com>

          * changed licenses to MIT
          * fixed typo in LICENSE
          * contact suffices
          * fixed syckle config
          * update tasks and metadata


  2009-10-06   7rans <transfire@gmail.com>

          * reap became syckle


  2009-08-17   proutils <proutils@tigerops.org>

          * first commit

  </pre>

  <p>A variety of formats are available, here are examples of the <a href="exmple.xml">XML</a>, <a href="example.html">HTML</a>,
  <a href="example.yaml">YAML</a> and <a href="example.json">JSON</a> formats.

  <p>One ot the coolest features of vcLog is it's ability to lookup tags and build a Release History file.</p>

  <pre>
  $ vclog --rel --ver 1.1.0

  == 1.1.0 / 2009-10-23

  * 1 General Enhancements

      * update pom metadata


  == 1.0.0 / 2009-10-13

  * 6 General Enhancements

      * reap became syckle
      * update tasks and metadata
      * fixed syckle config
      * contact suffices
      * fixed typo in LICENSE
      * changed licenses to MIT


  == 0.1.0 / 2009-08-17

  * 1 General Enhancements

      * first commit
  </pre>

  <p>Finally, vcLog recognizes commit types. There are single-word markers at the end of a commit messages wrapped in square brackets.
  It you utilize these, eg. <code>git commit -m "fixed foo issue [bug]"</code>, then you can sort the changelog output by type.
  You can actually see this already in the previous example. No types were used so it labled all the commits as "General Enhancements".
  If say a <code>[bug]</code> type were used it would have separated that commit out under <code>1 Bug Fixes</code>. vcLog recorgnizes
  <code>[maj]</code> or <code>[major]</code>, <code>[min]</code> or <code>[minor]</code>, <code>[bug]</code> and <code>[admin]</code> or <code>[-]</code>.
  Any other label is treated verbatim, ie. <code>[doozy]</code> would show up as "<code>1 Doozy Enhancement</code>".</p>

  <p>Commit types can be very useful to way to categorize your commits and thus present more readable changelogs --though it would be even nicer
  if SCMs supported them directly, e.g. with a <code>--type</code> option.</p>
